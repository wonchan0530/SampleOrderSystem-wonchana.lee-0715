# PRD - 반도체 시료 생산주문관리 시스템

> 출처: `docs/[CRA_AI] Day3_개인과제_반도체시료관리.pdf` (4~26페이지)

## 1. 배경

가상의 반도체 회사 **"S-Semi"**는 다양한 종류의 반도체 시료(Sample)를 생산하여 연구소, 팹리스(Fabless) 업체, 대학 연구실 등의 고객에게 납품한다. 시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고, 검수를 거쳐 고객에게 출고된다.

최근 주문량이 급증하면서 다음과 같은 문제가 발생했다.
- 주문 처리 여부를 파악하기 어렵다.
- 공정 예약 후 완성 시점을 알 수 없다.
- 재고가 충분한데도 불필요한 추가 공정이 발생한다.

엑셀과 메모장으로 주문을 관리하다 보니 실수가 잦고, 재고와 공정 현황을 한눈에 파악하기 어려운 것이 근본 원인이다. 이를 해결하기 위해 더 체계적인 시료 관리를 위한 **"반도체 시료 생산주문관리 시스템"**을 개발한다.

## 2. 시스템 개요

- 시스템은 **콘솔(CLI) 기반**으로 동작하며, 담당자가 직접 명령을 입력해 시료를 등록하고 주문을 처리한다.
- **생산라인**은 공장에서 시료 하나를 생산하는 설비 흐름이다.
  - 하나의 생산라인은 시료를 한 번에 하나씩 생산한다.
  - 주문이 들어온 시료에 대해서만 생산한다.

## 3. 사용자(역할)

- **주문 담당자**: 고객 주문을 접수하고 관리한다.
- **생산라인 담당자**: 접수된 주문을 승인·거절하고, 생산 현황을 관리한다.
- 두 역할 모두 동일한 콘솔 애플리케이션을 통해 각자 필요한 메뉴를 사용한다고 가정한다(역할별 로그인/권한 분리는 명세에 없음).

## 4. 도메인 용어

### 4.1 시료 (Sample)
시스템의 가장 기본이 되는 단위. 시스템에 등록된 시료만 주문할 수 있다.

| 속성 | 설명 |
|---|---|
| 시료 ID | 시료를 식별하는 고유 값 |
| 이름 | 시료 이름 |
| 평균 생산시간 | 시료 1개를 생산하는 데 걸리는 평균 시간 |
| 수율 | 정상적으로 생산된 시료 비율 = 정상 생산 시료 수 / 총 생산 시료 수 (예: 100개 생산 중 정상품 90개 → 수율 0.9) |

### 4.2 주문 (Order)
고객이 특정 시료를 원하는 수량만큼 요청하는 단위. 주문은 아래 상태 중 하나를 가진다.

| 상태 | 의미 |
|---|---|
| RESERVED | 주문 접수(초기 상태) |
| REJECTED | 주문 거절. 정상 흐름 밖의 상태이며 모니터링 대상에서 제외 |
| PRODUCING | 주문 승인 완료 + 재고 부족으로 생산 중 |
| CONFIRMED | 주문 승인 완료 + 출고 대기 중 |
| RELEASE | 출고 완료 |

### 4.3 주문 상태 전이

```
RESERVED --(승인, 재고 충분)--> CONFIRMED --(출고)--> RELEASE
RESERVED --(승인, 재고 부족)--> PRODUCING --(생산 완료)--> CONFIRMED --(출고)--> RELEASE
RESERVED --(거절)--> REJECTED
```

## 5. 기능 요구사항

### 5.1 메인 메뉴
- 기능별 선택 화면을 제공한다.
- 전체 시료에 대한 요약 정보를 확인할 수 있어야 한다.
- 하위 메뉴: 시료관리 / 주문(접수·승인·거절) / 모니터링 / 출고처리 / 생산라인

### 5.2 시료관리
- **시료 등록**: 새로운 시료를 시스템에 추가한다. 입력값은 시료 ID, 이름, 평균 생산시간, 수율이다.
- **시료 조회**: 등록된 모든 시료 목록을 확인한다. 각 시료의 현재 재고 수량도 함께 표시한다.
- **시료 검색**: 이름 등 속성으로 특정 시료를 검색한다.
- 등록되지 않은 시료는 주문할 수 없다.

### 5.3 시료 주문 (접수)
- 고객이 시료를 요청하면 주문 담당자가 주문을 생성한다.
- 입력값: 시료 ID, 고객명, 주문 수량
- 주문 생성 시점의 상태는 `RESERVED`이다.

### 5.4 주문 승인/거절
- 접수된 `RESERVED` 상태의 주문 목록을 확인할 수 있다.
- **주문 승인**: 특정 주문에 대해 승인 처리한다. 승인 시 재고 상황에 따라 자동으로 두 가지 방식 중 하나로 처리된다.
  - 재고가 충분한 경우 → 재고에서 주문 수량 전체를 즉시 차감하고, 주문을 즉시 `CONFIRMED` 상태로 전환
  - 재고가 부족한 경우 → 남아있는 재고를 0이 될 때까지 모두 이 주문에 차감(할당)하고, 부족분만 생산라인에 자동으로 등록, 주문 상태를 `PRODUCING`으로 전환
  - **재고 확인·차감은 이 승인 시점에만 이루어진다.** 이후 생산이 완료되어도 재고 확인/차감 로직은 다시 실행되지 않고, 생산 결과를 반영(가산)하기만 한다. 자세한 내용은 §6 참고.
- **주문 거절**: 특정 주문에 대해 거절 처리하면 즉시 `REJECTED` 상태로 전환된다.

### 5.5 모니터링
- 담당자가 시스템의 현재 상태를 한눈에 파악할 수 있도록 구성한다.
- **주문량 확인**: 현재 상태별(`RESERVED` / `CONFIRMED` / `PRODUCING` / `RELEASE`) 주문 수를 확인한다. `REJECTED`는 유효한 주문이 아니므로 집계에서 제외한다.
- **재고량 확인**: 시료별 현재 재고 수량을 확인한다. 주문 대비 재고 수량에 따라 아래 상태로 표기한다.
  - **여유**: 주문 대비 재고 충분
  - **부족**: 주문 대비 재고 수량 부족
  - **고갈**: 재고 수량이 0

### 5.6 출고처리
- 재고가 충분해진 `CONFIRMED` 상태 주문에 대해 출고를 처리한다.
- 특정 주문을 선택해 출고를 실행하면 주문 상태가 `RELEASE`로 전환된다.

### 5.7 생산라인
- 현재 생산 중인 시료 및 대기 중인 생산 큐(대기열)에 대한 정보를 표시한다.
- 주문량에 대한 부족분을 생산하되, 수율 및 오차를 고려하여 시료를 생산한다.
  - 실생산량 = `ceil(부족분 / 수율)`
  - 총생산시간 = `평균 생산시간 * 실생산량`
- 생산이 완료되면 해당 주문 상태는 `PRODUCING` → `CONFIRMED`로 전환된다.
- 생산 현황 표기: 현재 생산 중인 시료에 대한 정보를 표시한다(표기 정보 수준은 자유롭게 결정 가능, 예: 주문 정보, 현재까지의 생산량 등).
- 대기 주문 확인: 생산라인의 대기열(생산 큐)에서 생산 작업을 기다리고 있는 목록을 출력한다.
  - 스케줄링 전략: **FIFO** (선입선출)
- 생산 진행/완료 판정과 재시작 시 처리 방식은 §6을 따른다.

## 6. 재고 및 생산 정책 상세

이 섹션은 §5.4(주문 승인/거절), §5.7(생산라인)의 재고·생산 처리 방식을 구체화한다.

### 6.1 재고 차감은 승인 시점에만 발생

- 재고 확인과 차감은 **주문 승인 시점에 단 한 번**만 일어난다. 승인 이후(생산 진행 중 또는 완료 후)에는 재고 부족 여부를 다시 판단하지 않는다.
- 승인 시 재고가 충분하면 주문 수량 전체를 차감하고 `CONFIRMED`.
- 승인 시 재고가 부족하면, 남아있는 재고를 **0이 될 때까지 전부** 이 주문에 차감(할당)하고, 부족분(`주문 수량 - 승인 시점 재고`)만 생산 큐에 등록한다. 주문은 `PRODUCING`.
- 이 정책 때문에 **동일 시료에 대한 후속 주문**은 승인 시점에 이미 줄어든(또는 0이 된) 재고를 보게 되므로, 앞선 주문에 할당된 몫과 자연히 겹치지 않는다. (예: 재고 10에 주문A 수량 15가 승인되면 재고는 0이 되고 부족분 5가 생산 큐에 등록됨. 뒤이어 온 주문B는 재고 0을 기준으로 자신의 부족분 전체를 생산 큐에 등록한다.)

### 6.2 실생산량과 초과분 처리

- 실생산량 = `ceil(부족분 / 수율)`. 수율은 "필요한 부족분을 확실히 채우기 위해 얼마나 더 생산해야 하는가"를 계산하는 데만 쓰인다.
- **실제로 생산되는 수량은 모두 정상품으로 간주한다** — 즉 실생산량 전량이 사용 가능한 것으로 취급되며 불량으로 인한 손실은 없다고 가정한다.
  - 예: 부족분 60, 수율 0.6 → 실생산량 = `ceil(60/0.6)` = 100개. 실제로 100개 모두 정상 생산된다.
- 생산이 완료되면: 부족분만큼(위 예시에서 60개)은 해당 주문에 배정되어 `PRODUCING` → `CONFIRMED`로 전환되고, 나머지(실생산량 - 부족분, 위 예시에서 40개)는 재고에 새로 쌓인다.
- 즉 생산 완료 시점에 하는 일은 "재고 확인/차감 재판단"이 아니라, **이미 정해진 부족분을 주문에 채우고 초과분만 재고에 더하는 것**뿐이다.

### 6.3 생산 진행/완료 판정 — 실제 시스템 시각(벽시계) 기준

- 생산 큐의 각 작업은 시작 시각(`startedAt`)과 총생산시간(`totalProductionTime`, §5.7 계산식)을 가진다. 완료 예정 시각은 `startedAt + totalProductionTime`이다.
- 완료 여부는 **실제 시스템 시각**을 기준으로 판단한다. 즉 "현재 시각 ≥ 완료 예정 시각"이면 해당 작업은 완료된 것으로 간주하고, 다음과 같은 시점에 완료 처리를 수행한다.
  - 사용자가 관련 메뉴(생산라인, 모니터링, 출고처리 등)에 진입/조회할 때
  - 프로그램을 재시작해 큐/주문 데이터를 다시 로드할 때
- 생산라인은 한 번에 하나의 작업만 진행하므로(§5.7), 완료 판정 시 큐의 **맨 앞 작업부터 순서대로** 검사한다. 맨 앞 작업이 완료 조건을 만족하면 완료 처리 후(§6.2) 큐에서 제거하고, 다음 작업의 시작 시각을 "직전 작업의 완료 예정 시각"으로 설정한 뒤 다시 완료 여부를 검사한다 — 이 과정을 완료 조건을 만족하지 않는 작업을 만날 때까지 반복한다.
  - 이렇게 체인으로 검사하는 이유: 프로그램이 오래 종료되어 있던 동안 여러 작업이 연달아 완료 시각을 지났을 수 있기 때문이다.
- **프로그램이 종료되어 있던 동안 진행된 생산도 실제 경과 시간만큼 반영되어야 한다.** 따라서 생산 큐(각 작업의 주문 ID/시료 ID/부족분/실생산량/시작 시각/총생산시간)는 파일 등으로 영속화되어, 재시작 시 이를 다시 불러와 위 완료 판정 로직을 그대로 적용해야 한다.

## 7. 비기능 요구사항 / 제약사항

- 인터페이스는 콘솔(CLI) 기반이어야 한다. (예시 UI 화면은 참고용이며 실제 화면 구성은 자유롭게 결정)
- 데이터는 애플리케이션을 재실행해도 유지되어야 한다(데이터 영속성). 생산 큐의 진행 상태(§6.3)도 예외 없이 영속화 대상이다.
- 빌드/실행 환경은 MSBuild/Visual Studio 기반이며, CMake 등 크로스플랫폼 빌드 시스템은 사용하지 않는다.

## 8. 범위 밖 / 미정 사항

- 역할별(주문 담당자 vs 생산라인 담당자) 로그인·권한 분리 여부는 명세에 없음 — 필요 시 추가 논의 후 확정한다.
- 화면(UI) 레이아웃은 예시일 뿐이며 자유롭게 결정한다.
- 데이터 저장 방식(파일/JSON/DB 등) 및 상세 아키텍처, 개발 프로세스(테스트, Harness 등)는 `CLAUDE.md`를 참고한다.
