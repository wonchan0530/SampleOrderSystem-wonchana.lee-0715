# PRD - 반도체 시료 생산주문관리 시스템

> 출처: `docs/[CRA_AI] Day3_개인과제_반도체시료관리.pdf` (4~26페이지)

## 1. 배경

가상의 반도체 회사 **"S-Semi"**는 다양한 종류의 반도체 시료(Sample)를 생산하여 연구소, 팹리스(Fabless) 업체, 대학 연구실 등의 고객에게 납품한다. 시료는 주문이 들어오면 웨이퍼 공정 설비를 통해 제작되고, 검수를 거쳐 고객에게 출고된다.

최근 주문량이 급증하면서 다음과 같은 문제가 발생했다.
- 주문 처리 여부를 파악하기 어렵다.
- 공정 예약 후 완성 시점을 알 수 없다.
- 재고가 충분한데도 불필요한 추가 공정이 발생한다.

엑셀과 메모장으로 주문을 관리하다 보니 실수가 잦고, 재고와 공정 현황을 한눈에 파악하기 어려운 것이 근본 원인이다. 이를 해결하기 위해 더 체계적인 시료 관리를 위한 **"반도체 시료 생산주문관리 시스템"**을 개발한다.

## 2. 시스템 개요

- 시스템은 **콘솔(CLI) 기반**으로 동작하며, 담당자가 직접 명령을 입력해 시료를 등록하고 주문을 처리한다.
- **생산라인**은 공장에서 시료 하나를 생산하는 설비 흐름이다.
  - 하나의 생산라인은 시료를 한 번에 하나씩 생산한다.
  - 주문이 들어온 시료에 대해서만 생산한다.

## 3. 사용자(역할)

- **주문 담당자**: 고객 주문을 접수하고 관리한다.
- **생산라인 담당자**: 접수된 주문을 승인·거절하고, 생산 현황을 관리한다.
- 두 역할 모두 동일한 콘솔 애플리케이션을 통해 각자 필요한 메뉴를 사용한다고 가정한다(역할별 로그인/권한 분리는 명세에 없음).

## 4. 도메인 용어

### 4.1 시료 (Sample)
시스템의 가장 기본이 되는 단위. 시스템에 등록된 시료만 주문할 수 있다.

| 속성 | 설명 |
|---|---|
| 시료 ID | 시료를 식별하는 고유 값 |
| 이름 | 시료 이름 |
| 평균 생산시간 | 시료 1개를 생산하는 데 걸리는 평균 시간 |
| 수율 | 정상적으로 생산된 시료 비율 = 정상 생산 시료 수 / 총 생산 시료 수 (예: 100개 생산 중 정상품 90개 → 수율 0.9) |

### 4.2 주문 (Order)
고객이 특정 시료를 원하는 수량만큼 요청하는 단위. 주문은 아래 상태 중 하나를 가진다.

| 상태 | 의미 |
|---|---|
| RESERVED | 주문 접수(초기 상태) |
| REJECTED | 주문 거절. 정상 흐름 밖의 상태이며 모니터링 대상에서 제외 |
| PRODUCING | 주문 승인 완료 + 재고 부족으로 생산 중 |
| CONFIRMED | 주문 승인 완료 + 출고 대기 중 |
| RELEASE | 출고 완료 |

### 4.3 주문 상태 전이

```
RESERVED --(승인, 재고 충분)--> CONFIRMED --(출고)--> RELEASE
RESERVED --(승인, 재고 부족)--> PRODUCING --(생산 완료)--> CONFIRMED --(출고)--> RELEASE
RESERVED --(거절)--> REJECTED
```

## 5. 기능 요구사항

### 5.1 메인 메뉴
- 기능별 선택 화면을 제공한다.
- 전체 시료에 대한 요약 정보를 확인할 수 있어야 한다.
- 하위 메뉴: 시료관리 / 주문(접수·승인·거절) / 모니터링 / 출고처리 / 생산라인

### 5.2 시료관리
- **시료 등록**: 새로운 시료를 시스템에 추가한다. 입력값은 시료 ID, 이름, 평균 생산시간, 수율이다.
- **시료 조회**: 등록된 모든 시료 목록을 확인한다. 각 시료의 현재 재고 수량도 함께 표시한다.
- **시료 검색**: 이름 등 속성으로 특정 시료를 검색한다.
- 등록되지 않은 시료는 주문할 수 없다.

### 5.3 시료 주문 (접수)
- 고객이 시료를 요청하면 주문 담당자가 주문을 생성한다.
- 입력값: 시료 ID, 고객명, 주문 수량
- 주문 생성 시점의 상태는 `RESERVED`이다.

### 5.4 주문 승인/거절
- 접수된 `RESERVED` 상태의 주문 목록을 확인할 수 있다.
- **주문 승인**: 특정 주문에 대해 승인 처리한다. 승인 시 재고 상황에 따라 자동으로 두 가지 방식 중 하나로 처리된다.
  - 재고가 충분한 경우 → 주문을 즉시 `CONFIRMED` 상태로 전환
  - 재고가 부족한 경우 → 생산라인에 자동으로 등록하고, 주문 상태를 `PRODUCING`으로 전환
- **주문 거절**: 특정 주문에 대해 거절 처리하면 즉시 `REJECTED` 상태로 전환된다.

### 5.5 모니터링
- 담당자가 시스템의 현재 상태를 한눈에 파악할 수 있도록 구성한다.
- **주문량 확인**: 현재 상태별(`RESERVED` / `CONFIRMED` / `PRODUCING` / `RELEASE`) 주문 수를 확인한다. `REJECTED`는 유효한 주문이 아니므로 집계에서 제외한다.
- **재고량 확인**: 시료별 현재 재고 수량을 확인한다. 주문 대비 재고 수량에 따라 아래 상태로 표기한다.
  - **여유**: 주문 대비 재고 충분
  - **부족**: 주문 대비 재고 수량 부족
  - **고갈**: 재고 수량이 0

### 5.6 출고처리
- 재고가 충분해진 `CONFIRMED` 상태 주문에 대해 출고를 처리한다.
- 특정 주문을 선택해 출고를 실행하면 주문 상태가 `RELEASE`로 전환된다.

### 5.7 생산라인
- 현재 생산 중인 시료 및 대기 중인 생산 큐(대기열)에 대한 정보를 표시한다.
- 주문량에 대한 부족분을 생산하되, 수율 및 오차를 고려하여 시료를 생산한다.
  - 실생산량 = `ceil(부족분 / 수율)`
  - 총생산시간 = `평균 생산시간 * 실생산량`
- 생산이 완료되면 해당 주문 상태는 `PRODUCING` → `CONFIRMED`로 전환된다.
- 생산 현황 표기: 현재 생산 중인 시료에 대한 정보를 표시한다(표기 정보 수준은 자유롭게 결정 가능, 예: 주문 정보, 현재까지의 생산량 등).
- 대기 주문 확인: 생산라인의 대기열(생산 큐)에서 생산 작업을 기다리고 있는 목록을 출력한다.
  - 스케줄링 전략: **FIFO** (선입선출)

## 6. 비기능 요구사항 / 제약사항

- 인터페이스는 콘솔(CLI) 기반이어야 한다. (예시 UI 화면은 참고용이며 실제 화면 구성은 자유롭게 결정)
- 데이터는 애플리케이션을 재실행해도 유지되어야 한다(데이터 영속성).
- 빌드/실행 환경은 MSBuild/Visual Studio 기반이며, CMake 등 크로스플랫폼 빌드 시스템은 사용하지 않는다.

## 7. 범위 밖 / 미정 사항

- 역할별(주문 담당자 vs 생산라인 담당자) 로그인·권한 분리 여부는 명세에 없음 — 필요 시 추가 논의 후 확정한다.
- 화면(UI) 레이아웃은 예시일 뿐이며 자유롭게 결정한다.
- 데이터 저장 방식(파일/JSON/DB 등) 및 상세 아키텍처, 개발 프로세스(테스트, Harness 등)는 `CLAUDE.md`를 참고한다.
