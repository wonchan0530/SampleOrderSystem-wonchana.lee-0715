# 생산라인

> 상위 문서: [`../PRD.md`](../PRD.md) §5.7, §2

## 개요

주문 승인 시 재고가 부족하면 자동으로 생산라인 큐에 등록되어 부족분을 생산한다. 하나의 생산라인은 한 번에 시료 하나를 생산하며, 대기 중인 생산 작업은 **FIFO(선입선출)** 로 처리된다.

## 핵심 계산식

| 항목 | 계산식 |
|---|---|
| 부족분 | `주문 수량 - 현재 재고 수량` (주문 승인 시점 기준, 1회만 계산) |
| 실생산량 | `ceil(부족분 / 수율)` — 필요한 부족분을 확실히 채우기 위해 필요한 만큼보다 더 많이 생산 |
| 총생산시간 | `평균 생산시간 * 실생산량` |

예시: 부족분 5, 수율 0.9 → 실생산량 = `ceil(5 / 0.9)` = `ceil(5.56)` = **6개**. 평균 생산시간이 2(단위)라면 총생산시간 = `2 * 6` = **12**.

### 실생산량은 전량 정상품으로 취급 (초과분은 재고로)

- 수율은 "부족분을 확실히 채우려면 얼마나 더 생산해야 하는가"를 계산하는 데만 사용한다. **실제로 생산되는 실생산량은 수율과 무관하게 전량 사용 가능(정상품)한 것으로 가정한다.** 불량으로 인한 손실은 없다.
- 예: 부족분 60개, 수율 0.6 → 실생산량 = `ceil(60/0.6)` = **100개**. 이 100개는 모두 정상적으로 생산된다.
- 생산 완료 시 처리:
  1. 실생산량 중 **부족분만큼**(위 예시: 60개)은 해당 주문에 배정되어 재고 차감 없이 바로 주문을 충족시킨다.
  2. **나머지(실생산량 - 부족분, 위 예시: 40개)는 해당 시료의 재고에 새로 더해진다.**
- 즉 생산 완료 처리는 "재고를 다시 확인해서 충분한지 판단"하는 게 아니라, **이미 정해진 부족분을 주문에 채우고 초과 생산분만 재고에 가산**하는 단순 반영 작업이다. 재고 확인/차감 판단은 오직 [`03-주문관리.md`](03-주문관리.md)의 주문 승인 시점에만 일어난다.

## 생산 큐 (대기열)

- 주문 승인 시 재고 부족으로 `PRODUCING` 전환된 주문은 생산 큐에 순서대로 등록된다.
- **스케줄링 전략: FIFO** — 먼저 등록된 생산 작업이 먼저 처리된다.
- 한 번에 하나의 생산 작업만 실제로 "생산 중" 상태가 되며, 나머지는 대기 상태로 큐에 남는다.

## 하위 기능

### 1) 생산 현황 표기
- 현재 생산 중인 시료에 대한 정보를 표시한다.
- 표시 수준은 자유롭게 결정 가능하나 최소한 다음을 포함하는 것을 권장한다.
  - 생산 중인 시료명/주문 정보
  - 실생산량, 총생산시간
  - 현재까지의 생산 진행량(또는 남은 시간)

### 2) 대기 주문 확인
- 생산 큐에서 대기 중인 생산 작업 목록을 FIFO 순서대로 출력한다.
- 표시 항목(예): 대기 순번, 주문 ID, 시료명, 실생산량

### 3) 생산 완료 판정 — 실제 시스템 시각(벽시계) 기준

- 생산 큐에 등록되는 각 작업은 다음 정보를 갖는다: 주문 ID, 시료 ID, 부족분, 실생산량, 총생산시간, **시작 시각(`startedAt`)**. 완료 예정 시각 = `startedAt + 총생산시간`.
- 완료 여부는 **실제 시스템 시각**으로 판단한다: `현재 시각 ≥ 완료 예정 시각`이면 해당 작업은 완료된 것으로 간주한다. 타이머나 별도 스레드로 감시하지 않고, 아래 시점에 판정을 수행한다(지연 평가).
  - 생산라인/모니터링/출고처리 등 관련 메뉴에 진입하거나 조회할 때
  - 프로그램을 재시작해 데이터를 다시 불러올 때
- 생산라인은 한 번에 하나의 작업만 진행하므로, 완료 판정은 **큐 맨 앞 작업부터** 검사한다.
  1. 맨 앞 작업이 완료 조건을 만족하면 완료 처리(§ 위 "실생산량은 전량 정상품으로 취급" 절차)를 수행하고 큐에서 제거한다.
  2. 다음 작업이 있으면, 그 작업의 시작 시각을 "직전 작업의 완료 예정 시각"으로 설정하고 다시 완료 여부를 검사한다.
  3. 맨 앞 작업이 아직 완료 조건을 만족하지 않으면 검사를 멈춘다.
  - 여러 작업을 체인으로 검사하는 이유: 프로그램이 오래 종료되어 있던 동안 두 개 이상의 작업이 연달아 완료 시각을 지났을 수 있기 때문이다.
- 각 작업이 완료되면: 해당 주문 상태를 `PRODUCING` → `CONFIRMED`로 전환하고, 초과 생산분을 재고에 가산한다.

### 4) 프로그램 재시작 시 처리 (데이터 영속성)

- **프로그램이 종료되어 있던 동안 실제로 흐른 시간만큼 생산 진행도 그대로 인정된다.** 예를 들어 총생산시간이 남아있는 상태로 종료했다가, 총생산시간이 다 지난 뒤 재시작하면 그 즉시 생산 완료로 처리되어야 한다.
- 이를 위해 생산 큐의 각 작업(주문 ID/시료 ID/부족분/실생산량/총생산시간/시작 시각)은 **파일 등으로 영속화**되어야 하며, 재시작 시 이를 다시 불러온 뒤 위 "생산 완료 판정" 로직을 그대로 적용해 밀린 완료 처리를 수행한다.
- 재고/주문 상태(§03-주문관리.md)뿐 아니라 생산 큐 자체의 진행 상태도 영속화 대상이라는 점에 유의한다 — 큐가 비영속이면 재시작 시 진행 중이던 생산 정보(부족분, 시작 시각 등)를 잃어버려 이 요구사항을 만족할 수 없다.

## 예외/유효성 처리

| 상황 | 처리 |
|---|---|
| 생산 큐가 비어있는 상태에서 대기 주문 확인 | "대기 중인 생산 작업 없음" 표시 |
| 생산 중인 작업이 없는 상태에서 생산 현황 조회 | "현재 생산 중인 시료 없음" 표시 |
