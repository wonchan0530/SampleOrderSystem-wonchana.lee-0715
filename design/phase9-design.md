# Design — Phase 9: Clean Code 리뷰 및 문서/커밋 마무리

> 관련 계획: [`../PLAN.md`](../PLAN.md) Phase 9
> 상태: **승인됨 — 구현 완료** (Harness PASS — 4/4 빌드, 유닛테스트 87/87, E2E 시나리오)

> **점검 결과**: §2의 두 항목(Repository `recalcNextId`, Menu `run()` 골격) 모두 "실제 중복(Rule of Three/Five 충족)"으로 판단해 각각 `NextIdCalculator.h`, `MenuRunner.h`로 추출함. 리팩토링 전후 기존 테스트 87개가 그대로(수정 없이) 통과해 동작 무변경을 확인했다. §3의 문서 유보 문구 3건(`docs/02`, `docs/04`, `docs/05`)은 확정된 구현 내용으로 갱신함. 커밋 이력은 전부 `[feat]`/`[test]`/`[fix]`/`[etc]` 규칙을 준수하고 있어 별도 조치 없음.

## 1. 목표

"미션 2"의 나머지 주안점(Clean Code, 문서 관리, Commit 이력)을 마무리한다. 새 기능을 추가하지 않고, **기존 코드/문서를 실제 구현과 일치시키고 정리**하는 것이 전부다.

## 2. Clean Code 점검 — 실제 중복인지 판단

전체 소스를 다시 훑어 다음을 확인한다. 각 항목은 "문제 있음 → 수정" 또는 "의도된 설계 → 유지"로 결론을 낸다.

- **Repository들의 `recalcNextId`/`persist` 패턴 반복**(`SampleRepository`, `OrderRepository`, `ProductionQueueRepository`): 3곳에서 거의 동일한 보일러플레이트. → Phase 1 설계에서 "실제 중복이 확인되면 그때 추출"이라고 명시했고, 지금이 바로 그 시점(3곳 = Rule of Three 충족). **작은 헬퍼로 추출할지 검토** — 다만 `nextId_`의 "무엇의 최댓값을 찾는지"(id vs orderId)가 타입마다 달라 제네릭화하면 오히려 가독성이 떨어질 수 있어, 실제 리뷰에서 추출 여부를 판단한다(추출이 코드를 더 읽기 쉽게 만들 때만 진행).
- **각 `*Menu` 클래스의 `run()` 반복 구조**(메뉴 출력 → `readInt` → 분기 → `EofException` catch): 5개 메뉴에서 거의 동일한 골격. → Phase 2 설계에서 이미 "문구 차이를 보존하기 위해 의도적으로 추상화하지 않는다"고 결정했었나 재확인 필요 — 실제로는 Phase 2 설계에 이 판단이 명시적으로 기록되어 있지 않으므로, Phase 9에서 **다시 판단**한다. 결론 후보: 메뉴 항목 자체(제목/번호/핸들러)는 클래스마다 다르므로 완전한 템플릿화보다는 "그대로 유지"가 더 읽기 쉬울 가능성이 높지만, 코드를 다시 보고 실제로 결정한다.
- **Controller들의 `filterByStatus`류 헬퍼**: `OrderController`에 이미 있음(Phase 5). 다른 곳에 유사 패턴이 새로 생겼는지 확인.
- **미사용 include/헤더 정리**: 각 `.cpp`가 실제로 쓰지 않는 `#include`가 있는지 점검.
- **주석 검토**: 지금까지 작성한 주석이 "WHY"만 담고 있는지(현재 정책상 요구사항), "WHAT"을 반복 설명하는 불필요한 주석이 있으면 제거.

이 항목들은 실제 코드를 열어보며 판단하는 작업이라, 정확한 변경 목록은 구현(리뷰) 단계에서 확정한다 — 이 설계 문서는 "무엇을 점검할지"의 체크리스트 역할만 한다.

## 3. 문서 정합성 — 확정된 사양을 문서에 반영

구현하면서 확정된 내용인데 아직 문서에 "구현 시 결정" 같은 유보 문구로 남아있는 부분을 정리한다.

| 문서 | 현재 유보 문구 | 확정된 실제 구현으로 갱신 |
|---|---|---|
| `docs/05-출고처리.md` | "재고 차감 시점을 (a)/(b) 중 하나로 정해야 하며 ... 제안한다" | (a) 승인 시점 차감으로 확정됨(`OrderController::approve`) — 유보 문구 제거, 확정 사실로 서술 |
| `docs/04-모니터링.md` | "여유/부족/고갈 판정 기준은 ... 기본 규칙으로 제안한다" | `MonitoringController::classifyStockLevel`이 그대로 구현한 규칙으로 확정 — 유보 문구 제거 |
| `docs/02-시료관리.md` | "부분 일치(포함 검색) 지원 여부는 구현 시 결정" | `SampleRepository::search`가 이름 부분일치 + ID 완전일치를 지원하도록 확정됨 — 반영 |

- `CLAUDE.md`: 최종 프로젝트 구조(앱/테스트/DummyDataGenerator 3개 vcxproj), 최종 메인 메뉴 구성(1~5), Harness가 이제 E2E 시나리오까지 포함한다는 점(Phase 8)을 반영해 갱신.
- `PLAN.md`: Phase 0~9 전체가 완료되었음을 상단에 요약(각 Phase 링크 + 한 줄 결과)으로 표시.
- `PRD.md`: Phase 1~8을 거치며 실제 구현이 PRD와 어긋난 부분이 있는지 재확인(현재까지는 없음으로 파악되나, 재검토는 구현 단계에서 확정).

## 4. 커밋 이력 점검

- `git log --oneline`으로 전체 이력을 훑어 `[feat]`/`[test]`/`[fix]`/`[refac]`/`[etc]` 규칙을 어긴 커밋이 없는지 확인(이번 Phase에서 규칙 위반이 발견되면, 과거 커밋을 rewrite하지 않고 — 이력 재작성은 위험하므로 — 그대로 두고 앞으로만 준수).
- 각 Phase가 "설계 → 구현([feat]) → 테스트([test])" 최소 2커밋 이상으로 의미 있게 쪼개져 있는지 확인(이미 Phase 0~7에서 이 패턴을 지켜왔음).

## 5. 완료 기준 (Definition of Done)

- Clean Code 체크리스트(§2) 각 항목에 대해 "수정함" 또는 "유지 — 이유" 결론이 남음
- §3의 문서 정합성 갱신 완료, 실제 코드와 어긋나는 "구현 시 결정" 문구가 더 이상 없음
- `scripts/verify.ps1`(Phase 8에서 확장된 버전, 빌드+유닛테스트+E2E 시나리오) 최종 PASS
- 커밋 이력 점검 결과에 이상 없음(또는 발견된 사항을 기록)
